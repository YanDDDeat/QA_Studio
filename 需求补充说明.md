# 需求补充说明 — Pipeline数据流转设计修正

## 原设计错误说明

最初实现时，我对Pipeline数据流转方式理解有误，导致了以下错误：

1. **阶段2-6的输入方式错误**：将知识体系生成、问题校验、答案生成、答案校验、数据评估这5个阶段设计为"让用户逐条选择Dataset记录来执行任务"。这完全违背了需求规格——所有阶段都应该像问题生成一样，**选择一个JSON文件**作为输入，以文件为单位执行任务。

2. **数据存储方式错误**：将每条记录独立存入数据库的Dataset表，后续阶段通过查询Dataset表来获取数据。而需求规格明确指出："每个阶段的输出写回到同一个文件，在原记录上追加字段，不产生新文件（校验失败的除外）"。数据应以**JSON文件**为核心载体，数据库只是辅助。

3. **任务粒度错误**：每个任务应该是以**一个文件**为单位，处理文件中所有符合条件的记录。而不是让用户手动挑选哪些记录需要处理。

## 正确的Pipeline数据流转设计

### 核心原则

- **JSON文件是数据的核心载体**，数据库用于辅助查询和管理
- **所有阶段都选择JSON文件**作为输入，不是选择单条记录
- **一个任务 = 一个文件**，处理文件中全部符合条件的记录
- **结果覆盖写回同一个JSON文件**，逐步追加字段（校验失败除外）

### 流程模拟

以上传一个含2条text的源文本JSON为例：

#### 起点：上传源文本JSON

文件 `炸药知识.json`：
```json
[
  {"text": "RDX为白色粉末状结晶，熔点204℃...", "source": "单质与混合火工药剂"},
  {"text": "TNT为黄色结晶固体，密度1.654g/cm³...", "source": "单质与混合火工药剂"}
]
```

#### 阶段1：问题生成

- 用户操作：选文件、选category/source_type/提示词/模型，点开始
- 系统行为：逐条text调用LLM，1条text可能生成多个问答对，每个问答对成为一条独立记录
- **产出新JSON文件**（这是最终产物的起点，后续所有阶段都基于这个文件）

产出 `炸药知识_问题生成.json`：
```json
[
  {"id":1, "input":"RDX的熔点是多少？", "category":"知识问答", "task_type":"知识问答", "domain":"单质炸药", "difficulty":"较难", "corpus_cate":1, "source":"单质与混合火工药剂", "source_id":"...", "source_type":"图书", "originContent":"RDX为白色粉末状结晶，熔点204℃...", "Assessment":""},
  {"id":2, "input":"RDX在长期储存条件下应注意哪些温度控制？", ..., "Assessment":""},
  {"id":3, "input":"TNT的密度是多少？", ..., "Assessment":""}
]
```

#### 阶段2：知识体系生成

- 用户操作：选 `炸药知识_问题生成.json`，选提示词/模型，点开始
- 系统行为：逐条记录调用LLM生成knowledge，scene自动赋值=knowledge
- **覆盖写回**同一个文件，追加 `knowledge` 和 `scene` 字段

#### 阶段3：问题校验

- 用户操作：选同一个JSON文件，选提示词/模型，点开始
- 系统行为：逐条调用LLM校验，返回PASS/FAIL
- PASS的记录：**覆盖写回原文件**（不保留validation_result和reason字段）
- FAIL的记录：从原文件中剔除，**单独生成fail JSON文件**（保留源字段+validation_result+reason），存入文件管理区

#### 阶段4：答案生成

- 用户操作：选同一个JSON文件（只剩PASS记录），选提示词/模型，点开始
- 系统行为：逐条调用LLM（input+originContent），生成output和cot
- **覆盖写回**，追加 `output` 和 `cot` 字段

#### 阶段5：答案校验

- 同问题校验逻辑，PASS覆盖写回，FAIL单独生成fail文件

#### 阶段6：数据评估

- 用户操作：选同一个JSON文件，选提示词/模型，点开始
- 系统行为：逐条调用LLM评分
- **覆盖写回**，追加 `relevance, clarity, reasoning, terminology, score` 字段

#### 最终交付物

同一个JSON文件，每条记录所有字段完整：
```json
[
  {"id":1, "domain":"单质炸药", "category":"知识问答", "task_type":"知识问答", "input":"...", "output":"...", "cot":"...", "corpus_cate":1, "scene":"...", "Assessment":"", "source":"...", "source_id":"...", "source_type":"图书", "originContent":"...", "knowledge":"...", "difficulty":"较难", "relevance":4, "clarity":4, "reasoning":4, "terminology":4, "score":4.0}
]
```

### 任务日志持久化

当前问题：离开页面再回来，日志和进度全丢失。

修正方案：每个页面加载时，自动查询该阶段的最近任务。如果有运行中/已完成任务，自动恢复显示状态、进度条和日志。

### 数据库角色

数据库仍然用于：
- 存储用户信息、文件元数据、提示词版本、任务状态和日志
- Dataset表作为中间查询层（记录当前阶段、通过状态等），方便筛选哪些记录需要进入下一阶段
- 但**最终交付物是JSON文件**，每次阶段完成后必须同步写回文件
