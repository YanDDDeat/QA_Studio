# 需求补充说明 — Pipeline数据流转设计修正

## 原设计错误及原因

最初实现时对Pipeline数据流转方式理解有误，导致了三个核心错误：

### 错误1：阶段2-6让用户逐条选择Dataset记录
**错误做法**：知识体系生成、问题校验、答案生成、答案校验、数据评估这5个阶段，前端用多选表格让用户一条一条挑选Dataset记录来执行任务。

**原因**：我把"数据在各阶段间流转时，直接从数据库读取上一阶段的输出"理解成了"用户从数据库中挑选记录"，忽略了需求规格中"用户在前端选择'使用上一阶段的输出'作为输入"的含义。实际上，所有阶段的输入方式应该和问题生成一样——**选择一个JSON文件**作为输入，以文件为单位执行任务。

### 错误2：没有将结果写回JSON文件
**错误做法**：每条记录单独存入数据库的Dataset表，各阶段只更新数据库字段，不写回磁盘上的JSON文件。

**原因**：我过度依赖数据库作为唯一数据源，忽略了需求规格中明确要求"每个阶段完成后，输出写回同一个文件，在原记录上追加字段，不产生新文件（校验失败的除外）"。实际流程中，JSON文件才是用户的交付物，数据库只是中间工作副本。

### 错误3：任务日志不持久化
**错误做法**：前端页面只在当前会话内显示任务进度和日志，离开页面后状态丢失，回来时看不到之前的任务。

**原因**：前端组件的响应式状态没有在页面加载时自动恢复。应该查询该阶段的最近任务，自动恢复显示。

## 正确的Pipeline数据流转设计

### 核心原则
- **一个任务 = 一个JSON文件**
- **问题生成是起点，产出新JSON文件**（从源文本JSON转换而来）
- **后续5个阶段都选同一个JSON文件，每阶段完成后覆盖写回**
- **最终这个JSON文件的每条记录拥有所有字段，就是交付物**
- **校验失败的记录从文件中剔除，单独存到fail文件**

### 完整流程示例

#### 起点：上传源文本JSON
文件 `炸药知识.json`：
```json
[
  {"text": "RDX为白色粉末状结晶...", "source": "单质炸药", "source_id": "xxx_028"},
  {"text": "TNT为黄色结晶固体...", "source": "单质炸药", "source_id": "xxx_015"}
]
```

#### 阶段1：问题生成 — 产出新JSON文件
用户选源文件 + category + source_type + 提示词 + 模型 → LLM逐条处理text →
**产出新文件** `炸药知识_问题生成.json`，每条记录新增字段：id, input, category, task_type, domain, difficulty, corpus_cate, source, source_id, source_type, originContent, Assessment。一条text可能生成多个问题，每个问题成为一条独立记录。

#### 阶段2：知识体系生成 — 覆盖写回
用户选 `炸药知识_问题生成.json` + 提示词 + 模型 → LLM逐条处理 →
**覆盖写回**同一个文件，每条记录追加 knowledge, scene 字段。

#### 阶段3：问题校验 — 覆盖写回 + 生成fail文件
用户选同一个文件 + 提示词 + 模型 → LLM逐条校验 →
PASS的记录保留在文件中继续流转，FAIL的记录从文件中剔除。
**覆盖写回**通过的记录到原文件 + **单独生成**fail文件（包含源字段 + validation_result + reason）。

#### 阶段4：答案生成 — 覆盖写回
用户选同一个文件 + 提示词 + 模型 → LLM逐条生成 →
**覆盖写回**同一个文件，每条记录追加 output, cot 字段。

#### 阶段5：答案校验 — 覆盖写回 + 生成fail文件
同问题校验逻辑。

#### 阶段6：数据评估 — 覆盖写回
用户选同一个文件 + 提示词 + 模型 → LLM逐条评分 →
**覆盖写回**同一个文件，每条记录追加 relevance, clarity, reasoning, terminology, score 字段。

#### 最终交付物
同一个 `炸药知识_问题生成.json` 文件，每条记录所有字段完整填充，即为最终交付物。

## 需求补充2 — 文件选择不限制阶段 + 文件删除策略

### 文件选择不应限制阶段

**问题**：当前实现中，每个阶段的文件下拉只显示包含"上一阶段记录"的文件。比如知识体系生成只能选有question_generate阶段记录的文件，答案生成只能选有question_validate通过记录的文件。

**错误原因**：把流程限制得太死了。用户可能直接上传一个已经包含问题+knowledge的JSON文件，直接跳到答案生成环节。不应该强制要求必须完成上一阶段才能进入下一阶段。

**正确做法**：
- **所有阶段的文件下拉都显示用户的全部JSON文件**，不按阶段过滤
- 用户可以选择任意JSON文件进入任意阶段
- 执行时，系统只处理文件中**处于正确前序阶段的记录**，不符合条件的记录自动跳过
- 比如：用户直接上传一个已有问题字段的JSON进入知识体系生成 → 系统只处理有input字段但缺少knowledge字段的记录

### 文件删除策略修正

**问题**：当前删除文件时，如果任何Task记录引用了该文件（`Task.file_id`），就返回409拒绝删除，提示"Cannot delete file: 1 task(s) reference this file"。这太严格了——文件处理完成后Task也引用了它，用户想删文件却被阻止。

**正确做法**：
- 只有**正在运行中（status=running）**的Task引用的文件才阻止删除
- 已完成（completed）或已失败（failed）的Task引用的文件，允许正常删除
- 删除文件时，将引用该文件的Task的`file_id`设为NULL（而不是阻止删除）

## 需求补充3 — 侧边栏导航层级优化

**当前问题**：侧边栏只有一层平铺菜单，6个Pipeline阶段+3个管理页面全平铺展示，不够清晰。

**正确做法**：侧边栏改为分组折叠式，按业务逻辑分组：

1. **问题生成** → 问题生成 | 知识体系生成 | 问题校验
2. **答案生成** → 答案生成 | 答案校验
3. **数据评估** → 数据评估
4. **管理中心** → 配置中心 | 用户管理 | 数据中心

使用 Element Plus 的 `el-sub-menu` 实现可折叠的二级菜单，每个分组有图标和标题，子项为具体页面。
