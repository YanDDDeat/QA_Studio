# 需求 26：标注流水线2专业 CoT 构建

## 背景

现有 CoT/H-CoT 标注流程主要面向已有 CoT/H-CoT 节点化生成任务。本需求新增一个位于 COT/H-COT 标注分组下的页面入口，菜单名称为「标注流水线2」，用于从单篇完整论文自动构建专业 CoT 训练样本。

用户上传一个 JSON 文件，文件中仅包含一个 chunk，该 chunk 是一篇完整论文。用户指定论文正文所在字段名后，系统按固定 6 步提示词流水线自动生成当前支持的 10 类专业 CoT 数据。

当前支持的 CoT 类型包含以下 10 类：

1. 性能提升路径 CoT
2. 构效关系-结构性能关系 CoT
3. 候选分子 / 材料优选决策 CoT
4. 反事实结构改造 CoT
5. 失败原因诊断 CoT
6. 多目标约束优化 CoT
7. 机理到设计策略迁移 CoT
8. 实验条件 / 制备工艺优化 CoT
9. 实验方案生成 CoT
10. 实验设计配方 CoT

第一版不执行质检步骤，不写入 `cot_nodes`，不自动写入 `datasets`。

## 输入定义

### 源文件选择

标注流水线2的任务文件应优先从系统已有文件中选择，而不是只能在该页面临时上传。页面需要提供“选择已有文件”的主入口，必要时可保留“上传新文件”作为辅助入口，但创建流水线时应统一使用系统文件记录。

文件要求：

- 文件格式：JSON。
- 顶层结构：数组。
- 数组长度：必须大于等于 1。
- 数组中的每个元素代表一篇文献，且必须是对象。
- 用户需要指定论文正文所在字段名。
- 每个元素都必须包含该字段，字段值必须是非空字符串。
- `source` 字段可选；若存在，用作该文献的来源名称；若不存在，可使用数组序号或文件名生成来源标识。

单篇文献示例：

```json
[
  {
    "source": "paper1.md",
    "text": "完整论文正文...",
    "title": "...",
    "doi": "..."
  }
]
```

批量文献示例：

```json
[
  {
    "source": "paper1.md",
    "text": "文献1 Markdown 全文..."
  },
  {
    "source": "paper2.md",
    "text": "文献2 Markdown 全文..."
  }
]
```

批量 JSON 可由 CoT/H-CoT 标注分组下的「文本预处理」页面多 MD 合并功能生成。

### 正文字段选择

正文字段名不应固定为 `paper_text`。

第一版优化规则：

- 默认字段名使用 `text`。
- 更推荐：选择系统已有文件后，后端读取该 JSON 第一个对象的字段列表，前端以下拉框方式让用户选择正文字段。
- 若暂时无法自动读取字段列表，页面也应将默认值改为 `text`，用户可手动修改为 `content`、`paper_text` 等实际字段名。

页面参数：

- 系统已有 JSON 文件
- 正文字段名，默认 `text`，优先支持读取字段后选择
- LLM 配置 / 模型

### 单篇文献单 CoT 类型

标注流水线2的一次 run 可以包含多篇文献，但每篇文献只生成一个 CoT 类型。该类型不由用户手动选择，而是由该文献 Step 3 的 LLM 在 10 类 CoT 枚举中自动判定。

创建任务时，用户不需要选择目标 CoT 类型。系统会在一个 run 内串行处理 JSON 数组中的每篇文献；每篇文献独立执行 Step 1-6，并由该文献的 Step 3 自动选择一个最适合本篇文献的推荐类型。

规则：

- 每个文献 item 的 Step 3 必须输出 `recommended_cot_type`，该值必须来自当前支持的 10 类 CoT 枚举。
- 若某个 item 没有任何可构建类型，则该 item 标记为 skipped，并记录证据缺口或不可构建原因；其他 item 继续处理。
- 若某个 item 有多个类型可构建，模型必须选择证据链最完整、最适合生成单条训练样本的一类作为该 item 的 `recommended_cot_type`。
- 每个 item 只使用自己的 `recommended_cot_type` 执行 Step 4 / Step 5 / Step 6，不为同一 item 生成其他类型。
- 页面详情应展示整体批量进度，并能看到每篇文献的状态、来源、推荐 CoT 类型和错误/跳过原因。

## 流程规则

### Step 1：筛选相关文献

输入完整论文正文，判断该论文是否适合构建当前支持的 10 类专业 CoT。

输出要求：

- 是否可生成：`can_generate = true / false`
- 支持的 CoT 类型，仅允许从 10 类枚举中选择
- 不支持类型及原因
- 关键证据摘要

规则：

- 若 `can_generate = false`，流水线停止，并记录不可生成原因。
- 若 `can_generate = true`，继续 Step 2。

### Step 2：构建文献案例卡

输入完整论文正文，输出案例卡 `case_card`。字段包括：

- `source_id`
- `source_type`
- `material_or_molecule`
- `research_goal`
- `baseline`
- `modification_or_variable`
- `control_samples`
- `performance_metrics`
- `observed_results`
- `mechanism_claim`
- `process_conditions`
- `recipe_components`
- `validation_methods`
- `limitations`
- `failure_or_risk`
- `figures_or_tables`

无证据字段写 `null` 或 `[]`，不得补写。

### Step 3：自动判定本次任务的 CoT 类型

输入 Step 2 的 `case_card`，由 LLM 在 10 类 CoT 枚举中判断可构建类型，并自动选择一个推荐类型作为本次任务唯一生成类型。

输出要求：

- `constructible_cot_types`：可构建类型列表，每项包含 `cot_type`、`reason`、`evidence`
- `recommended_cot_type`：本次任务推荐生成的唯一 CoT 类型，必须来自 10 类枚举
- `recommendation_reason`
- `not_constructible_types`
- `missing_information`

若 `constructible_cot_types` 为空或 `recommended_cot_type` 为空，流水线停止并记录证据缺口。若存在多个可构建类型，后续 Step 4/5/6 只使用 `recommended_cot_type`。

### 总步骤定义

标注流水线2对每篇文献 item 都执行固定 6 个逻辑步骤：

1. Step 1：筛选相关文献
2. Step 2：构建文献案例卡
3. Step 3：自动判定本篇文献的 CoT 类型
4. Step 4：为 Step 3 推荐的 CoT 类型生成 input
5. Step 5：为 Step 3 推荐的 CoT 类型生成 chainofThought
6. Step 6：为 Step 3 推荐的 CoT 类型生成 output

注意：Step 4 / Step 5 / Step 6 只处理当前 item 的 Step 3 推荐类型。不得将 10 类 CoT 的 Step 4/5/6 全部提前展开为 30 个主步骤。批量输入时，页面总进度应以文献 item 为主，例如“文献 23/100，当前 Step 4”，而不是显示为 600 步。

### Step 4：生成 input

若 Step 3 判定存在推荐 CoT 类型，则读取该推荐类型对应的 Step 4 提示词，生成一个主 `selected_input`。

第一版每个任务只生成 1 条主样本，`alternative_inputs` 可以保留为中间结果，但不继续展开成额外样本。

### Step 5：生成 chainofThought

输入 Step 1-3 结果和推荐类型的 Step 4 结果，读取该推荐类型对应 Step 5 提示词生成 `chainofThought`。

训练文本必须去文献化，不得出现“根据文献”“本文报道”“作者发现”“图 1”“表 1”“Figure”“Table”等来源表达。证据来源仅保存在 `evidence_used` 或 `evidence_trace` 中。

### Step 6：生成 output

输入 Step 4 和 Step 5 结果，读取推荐类型对应 Step 6 提示词生成最终 `output`。

最终样本包含：

- `cot_type`
- `input`
- `chainofThought`
- `output`
- `evidence_trace`

## 文件化存储设计

第一版不使用 `cot_nodes`，不自动写入 `datasets`，每次流水线运行创建一个独立运行目录，所有阶段产物直接写 JSON 文件。

推荐目录结构：

```text
storage/
└─ professional_cot_runs/
   └─ <run_id>/
      ├─ manifest.json
      ├─ source.json
      ├─ final_samples.json
      ├─ final_samples.jsonl
      │
      ├─ item_0001/
      │  ├─ source_input.json
      │  ├─ step1_screening.json
      │  ├─ step2_case_card.json
      │  ├─ step3_target_judgement.json
      │  └─ <recommended_cot_type_key>/
      │     ├─ step4_input.json
      │     ├─ step5_chain.json
      │     └─ step6_output.json
      │
      ├─ item_0002/
      │  ├─ source_input.json
      │  ├─ step1_screening.json
      │  ├─ step2_case_card.json
      │  ├─ step3_target_judgement.json
      │  └─ <recommended_cot_type_key>/
      │     ├─ step4_input.json
      │     ├─ step5_chain.json
      │     └─ step6_output.json
      │
      └─ ...
```

单篇文献输入时也按 `item_0001/` 目录存储，保证单篇和批量输入的产物结构一致。

### manifest.json

作为一次运行的索引和进度文件，包含：

- `schema_version`
- `run_id`
- `pipeline_name = 标注流水线2`
- `pipeline_type = professional_cot`
- `status`
- `created_at`
- `updated_at`
- 上传源文件信息
- 正文字段名
- LLM 配置 / 模型
- 10 类 CoT 枚举
- steps 状态列表
- final output 文件路径
- error message

### 阶段产物文件

每个阶段产物直接保存对应 LLM JSON 输出，并额外保留必要元信息：

- `step`
- `step_name`
- `status`
- `cot_type` / `cot_type_key`，如适用
- 结构化结果
- `raw_llm_output`，如需要

### final_samples.json

汇总所有完成的 Step 6 输出：

```json
{
  "schema_version": "1.0",
  "run_id": "...",
  "pipeline_name": "标注流水线2",
  "sample_count": 1,
  "samples": [
    {
      "cot_type": "实验方案生成 CoT",
      "input": "...",
      "chainofThought": ["步骤 1：..."],
      "output": "...",
      "evidence_trace": {}
    }
  ]
}
```

同时生成 `final_samples.jsonl`，每行一个样本。

### 文件写入规则

- 阶段产物写入 JSON 文件。
- 写入时应先写临时文件，再原子替换目标文件，避免页面读取半截 JSON。
- 对外读取 artifact 时必须限制路径在当前 run 目录内，防止路径穿越。

### 页面风格参照

标注流水线2的页面风格必须参照现有标注流水线（`CotHcot/WorkflowList.vue` + `CotHcot/WorkflowDetail.vue`），保持一致的设计语言和交互模式：

- **列表页**：使用 `el-dialog` 弹窗新建任务 + `el-card` 包裹的 `el-table` 展示运行列表 + `el-tag` 展示状态 + `el-progress` 展示进度百分比 + 步数文字
- **详情页**：顶部 `el-card` 展示基本信息（`el-descriptions`）+ 总进度条卡片（圆点 tracker）+ 步骤卡片列表（圆圈序号、状态 tag、进度条、操作按钮）+ 文件预览 `el-dialog` + 运行中步骤自动轮询刷新
- **CSS**：沿用 `.page-container`、`.card-header`、`.step-card`、`.step-connector`、`.step-dot`、`.step-index` 等样式结构，颜色方案和尺寸保持一致

### 页面结构

标注流水线2采用两页结构（列表 + 详情），与现有标注流水线一致：

**1. 列表页（PipelineList.vue）**
- `el-dialog` 新建弹窗：
  - 选择系统已有 JSON 文件，上传新文件仅作为辅助入口
  - 选择或填写正文字段名，默认 `text`，优先从文件字段列表中选择
  - 选择 LLM 配置 / 模型（复用 `getLLMConfigs`）
  - 不提供目标 CoT 类型选择，CoT 类型由 Step 3 模型自动判定
  - 一键开始 Pipeline 按钮
- `el-card` + `el-table` 运行列表：
  - 显示 run id / 名称、文件名、状态、进度、创建时间
  - 支持「查看详情」跳转
  - 必须提供分页展示，支持切换页码和每页条数，避免 run 数量增多后列表过长

**2. 详情页（PipelineDetail.vue）**
- 顶部信息卡片（`el-descriptions`）：名称、文件名、正文字段名、模型、创建时间等；Step 3 完成后展示模型自动判定出的 CoT 类型
- 总进度条卡片：圆点 tracker 固定展示 6 个逻辑步骤
- 步骤卡片列表：
  - 创建 run 后必须立即展示固定 6 个步骤，运行中不得为空
  - 每个步骤显示圆圈序号、状态 tag、进度条、文件预览/下载
  - Step 3 未推荐可构建类型时，Step 4-6 标记为 skipped 或任务停止，并展示原因
- 文件预览弹窗：查看步骤 JSON 产物
- 运行中步骤自动轮询刷新
- 最终导出区：放在“流水线步骤”区域下方，下载 `final_samples.json` / `final_samples.jsonl`，并预览最终样本

## 后端接口建议

新增独立路由前缀：

```text
/api/professional-cot
```

建议接口：

- `POST /api/professional-cot/runs`：上传文件并启动一键 Pipeline
- `GET /api/professional-cot/runs`：分页读取运行列表，扫描 manifest，支持 `page` / `page_size` 参数，并返回 `items` / `total` / `page` / `page_size`
- `GET /api/professional-cot/runs/{run_id}`：读取运行详情和主要产物
- `GET /api/professional-cot/runs/{run_id}/artifact?path=...`：读取指定产物 JSON
- `GET /api/professional-cot/runs/{run_id}/export/json`：下载 `final_samples.json`
- `GET /api/professional-cot/runs/{run_id}/export/jsonl`：下载 `final_samples.jsonl`

第一版可以只支持启动、列表、详情、下载，不强制支持暂停/恢复。

## 验收标准

1. 侧边栏 COT/H-COT 标注分组下出现「标注流水线2」。
2. 页面优先支持选择系统已有单 chunk JSON 文件，上传新文件仅作为辅助入口。
3. 正文字段默认 `text`，优先支持读取文件字段列表后下拉选择。
4. 创建任务时不要求用户选择 CoT 类型；CoT 类型必须由 Step 3 模型自动判定。
5. 非法 JSON、数组长度不是 1、字段不存在、字段内容为空或字段不是字符串时给出错误提示。
6. 点击一键开始后，后端创建独立 run 目录并写入 `manifest.json`，manifest 中立即包含固定 6 个逻辑步骤。
7. 页面详情在任务运行中也必须展示 6 个步骤状态，不得等任务完成后才显示；步骤区域不得为空。
8. Step 1 判断不可生成时，流水线停止并在页面展示原因。
9. Step 1 可生成时，继续执行 Step 2、Step 3。
10. Step 3 从 10 类 CoT 中自动输出 `recommended_cot_type`；若没有推荐类型则停止并展示证据缺口。
11. 若存在 `recommended_cot_type`，仅对该推荐类型执行 Step 4、Step 5、Step 6，并分别写入 JSON 文件。
12. 最终生成 `final_samples.json` 和 `final_samples.jsonl`，每次任务默认 1 条样本。
13. 页面可分页查看运行记录，并支持切换页码和每页条数。
14. 页面可查看步骤结果和最终样本预览。
15. 最终产物区必须位于“流水线步骤”下方。
16. 页面可下载最终 JSON / JSONL。
17. 第一版不产生数据库迁移。
