# 需求：任务列表展示输入/输出文件名

## 背景

当前「我的任务」页面（`MyTasks.vue`）的任务列表有一列「输入文件」，但实际显示的内容并不准确：

- `Task` 表里有两个文件 ID 字段：`source_file_id`（输入）和 `file_id`（任务运行后会被**覆盖**为输出文件 ID）
- 后端 `tasks.py` 的 `list_tasks` / `my-running` 接口只 join 了 `file_id` 关联出 `filename`
- 前端把这个 `filename` 标成「输入文件」展示

**结果**：任务一旦运行，「输入文件」列实际显示的是**输出文件名**，列名误导；同时无法看到输入与输出的对照关系。

管理员视角（`ConfigCenter.vue` 系统设置标签页的「运行中任务」面板）也存在同样问题：完全没有文件名信息，admin 巡查时无法快速判断任务在处理什么文件、产出去了哪里。

## 目标

让任务列表清晰区分输入文件和输出文件：
- 列名不再误导
- 同时显示输入与输出（任务未启动时输出列显示 `-`）

## 范围

### 后端（1 个文件）

**`backend/app/routers/tasks.py`** 的 3 个接口都补字段：

#### 1. `GET /api/tasks` (`list_tasks`)
- 现有 join 改为同时查 `source_file_id` 和 `file_id` 关联的 filename
- 响应每条 item 新增两个字段：
  - `source_filename`: 输入文件名（来自 `source_file_id`，可能为 `null`）
  - `output_filename`: 输出文件名（来自 `file_id` 在任务运行后被改写的 output file，可能为 `null`）
- 保留旧字段 `filename`（向后兼容），值同 `output_filename`，避免老代码炸

#### 2. `GET /api/tasks/my-running` (`list_my_running_tasks`)
- 同样响应字段加 `source_filename` 和 `output_filename`
- 保留旧 `filename`

#### 3. `GET /api/tasks/running` (`list_running_tasks`，admin 用)
- 当前**完全没有**文件名字段，新增 `source_filename` 和 `output_filename`
- 同样批量查 file 表，避免 N+1

#### 实现要点
- 批量查 file 表（`File.id.in_(...)`），合并 `source_file_id` 和 `file_id` 的去重集合一次查完，构造 `{id: filename}` map 共用
- 任务刚创建（progress=0、output 文件还没建）时，`task.file_id` 可能等于 source_file_id 或为空，此时 `output_filename` 取出来跟 `source_filename` 一样 / 为 null 都可接受 —— 接口实现按字段实际值返回，不做特殊处理

### 前端（2 个文件）

#### `frontend/src/views/MyTasks.vue`
两个表格（运行中 + 历史）都改：
- 删除原「输入文件」列（min-width=160）
- 新增两列：
  - 「输入文件」：绑定 `row.source_filename`，min-width=160，空值显示 `-`
  - 「输出文件」：绑定 `row.output_filename`，min-width=160，空值显示 `-`
- 长文件名超出宽度时启用 `show-overflow-tooltip`，避免撑爆表格

#### `frontend/src/views/ConfigCenter.vue`
系统设置标签页「运行中任务」面板的表格，在「模型」列之后插入两列：
- 「输入文件」：`row.source_filename`，min-width=160，`show-overflow-tooltip`
- 「输出文件」：`row.output_filename`，min-width=160，`show-overflow-tooltip`

## 不涉及

- 不改数据库（`source_file_id` 字段已经存在并被填充）
- 不改 stage 各阶段 router 的逻辑
- 不改 `task.file_id` 在运行时被覆盖的行为（这是另一个独立问题，本次不动）
- 不动 `TaskConfigDialog.vue`、不动各阶段页面的状态恢复逻辑

## 兼容性

后端响应只**新增**字段，老字段 `filename` 保留不变（值 = `output_filename`），所以现有调用方（如阶段页面恢复逻辑、status 接口的下游消费）都不会受影响。

## 测试方案

人工：
1. 启动一个 question_generate 任务，进入 MyTasks 页面 → 运行中列表两列：输入文件应是上传的源文件，输出文件应是新生成的「xxx_问题生成_user_yyymmdd_HHMMSS.json」
2. 任务完成后进入历史列表 → 两列仍正确
3. admin 账号进配置中心系统设置标签 → 看到其他用户运行中任务的输入/输出文件
4. 一个刚创建还没开始跑的任务（progress=0）→ 输出文件列显示 `-` 而非报错
5. 历史里的旧任务（pre-feature 时代）若数据库字段缺失 → 列正常显示 `-`，不崩
