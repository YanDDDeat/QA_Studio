# QA_Studio 详细需求规格

## 1. 用户管理

1.1 管理员手动创建账号，无注册功能

1.2 无角色区分，每个用户只能看到自己的数据，数据完全隔离

1.3 简单的登录认证机制

---

## 2. 文件上传与管理

2.1 用户可上传一个或多个 JSON 文件

2.2 上传时需指定 JSON 中的文本块字段名，默认为 "text"

2.3 用户使用过程中生成、上传的所有文件，在工作区文件管理页面提供查看、下载、删除功能

2.4 校验阶段自动生成的失败记录文件，也在此页面查看/下载

2.5 每个阶段完成后，输出写回同一个文件，在原记录上追加字段，不产生新文件（校验失败的除外）

---

## 3. Pipeline 流程总览

系统包含 5 个独立阶段，用户按顺序在各阶段页面手动操作。每个阶段是独立任务，用户可灵活重复或跳步。数据在各阶段间流转时，直接从数据库读取上一阶段的输出，无需下载后重新上传。

阶段顺序：
- 3.1 问题生成
- 3.2 知识体系生成
- 3.3 问题校验
- 3.4 答案生成
- 3.5 答案校验
- 3.6 数据评估

---

## 4. 问题生成

4.1 用户操作：
- 选择一个已上传的 JSON 文件作为输入
- 选择问题类型 category，枚举值：["知识问答", "逻辑生成"]
- 选择 source_type，枚举值：["图书", "专利", "文献", "其他"]
- 可填写 source 和 source_id（若为空，从输入 JSON 的对应字段赋值，若无则使用上传的文件名）
- 选择提示词和模型
- 点击开始生成

4.2 生成逻辑：
- 读取 JSON 文件中每条记录的指定文本字段（默认 text）
- 逐条调用 LLM，LLM 返回一个包含多个问题的列表 [{},{}]
- 每条 text 可能生成多个问答对，每个问答对生成一条独立记录

4.3 产出字段：
| 字段 | 来源 |
|------|------|
| id | 系统全局唯一数字ID |
| input | LLM 生成后解析 |
| category | 用户选择 |
| task_type | LLM 生成后解析 |
| domain | LLM 生成后解析 |
| difficulty | LLM 生成后解析 |
| corpus_cate | 固定为 1 |
| source | 用户填写 > JSON字段 > 文件名 |
| source_id | 用户填写 > JSON字段 |
| source_type | 用户选择 |
| originContent | 来自输入 text |
| Assessment | 空字符串 |

4.4 进度展示：
- 显示进度条和当前已完成数量（如 "已完成 23/100 条 text，已生成 47 个问答对"）
- 实时更新

---

## 5. 知识体系生成

5.1 用户操作：
- 选择问题生成阶段的输出数据作为输入
- 选择提示词和模型
- 点击开始生成

5.2 生成逻辑：
- 逐条对每个问题调用 LLM，返回单个 JSON {}

5.3 产出字段：
| 字段 | 来源 |
|------|------|
| knowledge | LLM 生成后解析 |
| scene | 系统赋值，与 knowledge 字段值相同 |

---

## 6. 问题校验

6.1 用户操作：
- 选择知识体系生成阶段的输出数据作为输入
- 选择提示词和模型
- 点击开始生成

6.2 生成逻辑：
- 逐条对每个问题调用 LLM，返回格式：{ "validation_result": "PASS/FAIL", "reason": "原因说明" }
- 判断逻辑：validation_result == "PASS" 则保留，进入下一阶段
- 未通过的记录：自动生成单独的 JSON 文件，包含源字段 + validation_result + reason，存入文件管理区供用户查看/下载
- 通过的记录：不保留 validation_result 和 reason 字段，继续流转

6.3 进度展示同问题生成阶段

---

## 7. 答案生成

7.1 用户操作：
- 选择问题校验阶段的输出数据（仅通过的记录）作为输入
- 选择提示词和模型
- 点击开始生成

7.2 生成逻辑：
- 逐条对每个问题（input + originContent）调用 LLM

7.3 产出字段：
| 字段 | 来源 |
|------|------|
| output | LLM 生成后解析 |
| cot | LLM 生成后解析 |

---

## 8. 答案校验

8.1 用户操作：
- 选择答案生成阶段的输出数据作为输入
- 选择提示词和模型
- 点击开始生成

8.2 生成逻辑：
- 逐条对每个问答对调用 LLM，返回格式：{ "validation_result": "PASS/FAIL", "reason": "原因说明" }
- 判断逻辑同问题校验：PASS 保留，FAIL 写入单独文件到文件管理区
- 通过的记录：不保留 validation_result 和 reason 字段

8.3 进度展示同问题生成阶段

---

## 9. 数据评估

9.1 用户操作：
- 选择答案校验阶段的输出数据（仅通过的记录）作为输入
- 选择提示词和模型
- 点击开始生成

9.2 生成逻辑：
- 逐条对每个问答对调用 LLM

9.3 产出字段：
| 字段 | 来源 |
|------|------|
| relevance | LLM 生成后解析 |
| clarity | LLM 生成后解析 |
| reasoning | LLM 生成后解析 |
| terminology | LLM 生成后解析 |
| score | LLM 生成后解析 |

9.4 进度展示同问题生成阶段

---

## 10. 数据管理

10.1 表格展示每个数据集的记录

10.2 超长文本字段截断显示，超过限制字数显示 "..."

10.3 分页：10条/页

10.4 单击某条记录可查看详情

10.5 详情页每个字段支持 LaTeX 公式渲染

10.6 只展示当前用户自己的数据

---

## 11. 配置中心

11.1 每个 LLM 调用环节都可独立配置提示词

11.2 提示词修改时新增一个版本，保留历史版本

11.3 每个用户独立管理自己的提示词

11.4 每个环节可选择使用的模型

---

## 12. 任务日志

12.1 每个生成任务提供实时日志

12.2 前端只渲染最新 200 条日志

---

## 13. LLM 调用

13.1 采用 OpenAI 兼容接口（baseURL + API Key + model）

13.2 抽离为统一方法：输入提示词，返回 LLM 文本解析结果

13.3 逐条调用，不批量并发

13.4 多用户同时使用时各自独立调用，互不影响

---
