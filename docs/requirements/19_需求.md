# 需求：数据评分改名 + 质检页面 + 通用生成页面

## 背景

本次需求合并三个相关但独立的改动：

1. **改名**：用户希望把侧边栏的「数据评估」用词调整为「数据评分」，更贴合该阶段实际是给 QA 数据打综合评分（relevance / clarity / reasoning / terminology / score）的语义。
2. **新增质检页面**：评分之后还需要一道「质检」工序对评分结果再过一遍 LLM 复审，输出"是否通过 + 理由"，便于把不合格记录筛掉。
3. **新增通用生成页面**：现有 6 个生成/校验/评分页面在 UI 与数据流转上 80% 一致，差异只是 Prompt 内容和写入字段。需要一个开放沙盒页面，让用户能自由组合"任意文件 + 任意 Prompt + 任意 LLM"做一次性试跑或 Prompt 调试，LLM 返回什么字段就保存什么字段，不预设结构。

三项改动互不依赖，可独立交付，但都涉及侧边栏菜单与少量后端基础设施，合并到同一分支统一处理。

## 子需求 2：新增「质检」页面

### Pipeline 定位

```
... 数据评分 (data_evaluate) ──► 【质检 quality_check】 ──► COT过滤 / 切分 ...
```

- 默认输入：上一阶段 `data_evaluate` 的输出 JSON 文件
- 默认开关：FileSelector 的"是否展示全部文件"开关可让用户选任意 JSON（与现有 6 个阶段一致）
- **输出（双文件模式，与 `answer_validate` 完全一致）**：
  - **PASS 文件**：克隆通过的 dataset 到新输出文件 `xxx_质检_<用户名>_<时间戳>.json`（`source_stage='quality_check'`，`passed='是'`）
  - **Fail 文件**：未通过的记录汇总写入独立文件 `xxx_质检失败.json`（**不入 datasets 表**，只落盘），每条带 `validation_result='FAIL'` + `reason='...'` 字段

### 侧边栏菜单

**新建独立分组**（不挂在"数据评分"下）：

```
- 数据质检 (group-qc)
  └─ 质检   /quality-check
```

加在"数据评分"分组之后、"数据后处理"分组之前。

### 数据库变更（需要用户确认）

#### 零列变更 —— 完全照搬校验类模式

参考 `answer_validate` / `question_validate` 的处理：
- FAIL 记录根本不入 `datasets` 表（只在 fail 文件里）
- PASS 记录复用现有 `passed` 列写入 `"是"`，配合 `current_stage='quality_check'` 即可在数据中心筛选

**故无需新增任何列**，DB schema 完全保持不变。

#### `StageEnum` 枚举新增成员

```python
QUALITY_CHECK = "quality_check"
```

数据库层面需要 `ALTER TABLE` 修改 4 个 ENUM 列（`tasks.stage` / `prompts.stage` / `files.source_stage` / `datasets.current_stage`），把 `'quality_check'` 加入允许值（参考 `scripts/migrate_enum_extend.py` 既有模式）。

#### 默认 Prompt

迁移脚本中插入一条 `stage='quality_check'`、`user_id=NULL`（全局共享）、`is_default=True` 的默认 Prompt 模板，示例：

```
你是 QA 数据质检专家。根据给定记录的问题、答案、推理过程、知识体系以及评分维度，判断该条数据是否合格。

合格标准：综合评分 ≥ 4 且各维度评分 ≥ 3，且推理逻辑无明显错误。

请返回 JSON：
{
  "validation_result": "PASS" 或 "FAIL",
  "reason": "判断理由（中文，50-150 字）"
}
```

**注意**：LLM 返回字段命名与现有 `question_validate` / `answer_validate` 完全一致（`validation_result` + `reason`），便于复用前后端校验代码与 fail 文件格式。

### 迁移脚本

新建 `scripts/migrate_quality_check_and_generic.py`（与子需求 3 共用一个脚本），按现有 `migrate_enum_extend.py` + `migrate_default_prompts.py` 模式编写，包含：

1. 4 个 ENUM 列扩展 `'quality_check'` 和 `'generic'`（一次性加两个值）
2. 插入质检默认 Prompt（已存在则跳过）

执行前在文档中提示用户备份 DB；本次**不涉及 ALTER TABLE 加列**，只动 enum，迁移风险极低。

## 子需求 3：新增「通用生成页面」

### 设计意图

现有 6 个生成页面（问题生成/知识体系生成/问题校验/答案生成/答案校验/数据评分）的**功能本质相同**：
- 选文件 → 选 Prompt → 选 LLM → 调 LLM 处理每条记录 → 写回结果

差异只在 **Prompt 不同** 导致 **输出字段不同**。

通用生成页面定位为**开放沙盒**：用户可以任选一个文件、任选一个 Prompt（不限 stage）、任选 LLM 模型，跑一次性任务。LLM 返回的字段全部通过现有 `apply_llm_fields_to_dataset()` 自动映射：能匹配到 `datasets` 表列的写入对应列，无法匹配的统一进 `extra_fields` JSON 列。

**适用场景**：
- 临时跑一次新 Prompt 调试效果
- 复用某阶段的 Prompt 处理不属于该阶段的文件
- 用户自定义 Prompt（返回自定义字段）

### 侧边栏菜单

加入"工具"分组（和 JSON 合并、Gzip 上传并列）：

```
- 工具 (group-tools)
  ├─ JSON 文件合并
  ├─ Gzip 压缩上传
  └─ 通用生成   /generic-generate   ← 新增
```

### 数据库变更（需要用户确认）

`StageEnum` 新增成员：
```python
GENERIC = "generic"
```

理由：需要给输出文件打 `source_stage='generic'` 标签便于在数据中心识别；任务表也需要 `tasks.stage='generic'` 用于追踪。

迁移：复用上面质检的迁移脚本（同一次性扩展两个 enum 值）。

**不新增任何列** —— 完全复用 `extra_fields` JSON 列存储非标准字段。

### 后端改动

#### 新增文件

**`backend/app/routers/generic_generate.py`**：基于 `data_evaluate.py` 简化克隆，关键差异：

- 路由 prefix：`/api/generic_generate`
- `StartRequest`：
  - `file_id`、`prompt_id`、`model`、`llm_config_id`、`reference_fields`、`output_filename`（与现有阶段一致）
- `/start` 校验：
  - Prompt 不限 stage（去掉 `prompt_obj.stage != XXX` 校验）
  - 文件不限 source_stage
  - 不调用 `validate_file_fields()`（因为没有固定必填字段）
- `_run_generic_task()`：
  - LLM 调用、克隆 dataset、`apply_llm_fields_to_dataset()` 自动映射 → 输出文件
  - `source_stage=StageEnum.GENERIC`
  - 无定制化的"评分解析"逻辑（不像 data_evaluate 要解析 relevance/clarity 等）
- `/source-files`：**默认 `show_all=True`**（与其他阶段不同：通用页面不限上游）
- `/retry/{task_id}` 与 `resume_generic_task()`：参考 data_evaluate

#### 修改文件

**`backend/app/main.py`**：注册路由

**`backend/app/models/models.py`**：`StageEnum` 加 `GENERIC`

- **结果预览表格**：列由后端返回数据动态决定（categorizeFields 已支持 extra_fields 展开），无需固定列

## 实施顺序建议

1. 先做子需求 1（侧边栏改名）—— 1 个 commit，最低风险
2. 再做子需求 2（质检页面）—— 涉及 DB 迁移，需要用户先备份 DB 再执行迁移脚本

子需求 2、3 可在同一个分支同一个 PR 提交，因为迁移脚本一次跑完两者的 enum 扩展。
