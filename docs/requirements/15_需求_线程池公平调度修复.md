# 需求：线程池公平调度修复（SlidingWindowExecutor）

## 背景

`backend/app/services/thread_pool.py` 中的 `SlidingWindowExecutor` 用于按"活跃任务数"动态分配线程池配额（公式 `window_size = pool_size // active_task_count`），目的是让多个并发任务公平共享 40 个 worker。

但实际运行中发现严重不公平：先启动的任务（DYY）几乎独占线程池，后启动的任务（ZY、XF）速率骤降到 5%~0%。生产环境实测：

| 用户 | 启动时间 | 进度 | 估算速率 |
|---|---|---|---|
| DYY（最早） | 02:47 | 3605/5306 | ~2400/小时 |
| ZY（中间） | 03:27 | 142/638 | ~140/小时 |
| XF（最晚） | 04:21 | 0/2652 | ~0 |

期望（按 active=3 各占 ~13 worker）三人速率应接近 1:1:1。

## 问题根因

### Bug A：窗口缩放时新信号量"满血"

`acquire()` 中检测到 `window_size` 变化时直接 `self._sem = asyncio.Semaphore(ws)` 新建对象。新信号量初始有 `ws` 个 permit，相当于给当前用户额外"突击提交" `ws` 个调用的机会，绕过了限制。

```python
# thread_pool.py:108-110（问题代码）
if self._sem is None or ws != self._last_window_size:
    self._sem = asyncio.Semaphore(ws)   # 新建对象 → 满 permit
    self._last_window_size = ws
```

### Bug B：老 future 的回调释放到新信号量（permit 泄漏）

`track()` 注册的 done_callback 是 `lambda _: self._sem.release()`，`self._sem` 是晚绑定。窗口缩放后，老 future 完成时释放到的是**新信号量**，导致新信号量 permit 被超量灌入。

```python
# thread_pool.py:116（问题代码）
future.add_done_callback(lambda _: self._sem.release())
```

### 综合效果：snowball / starvation

- User A 在被 User B "打断"瞬间，可秒突击 ~20 个新提交
- 老 40 个 future 完成时全部释放 permit 到新信号量，User A 实际并发长期维持 ~40
- 线程池只有 40 worker，A 占满意味着 B 排到队尾
- 每当 B 完成一个调用，那个 worker 空出后会从 FIFO 队列头取下一个任务 → 队列头是 A 的提交 → worker 被 A 抢走
- B 的 worker 数指数衰减：`dY/dt ≈ -Y/10`，10s 后 B 剩 7 个，30s 后剩 1 个

## 输入 / 输出定义

- **输入**：N 个并发任务通过各自的 `SlidingWindowExecutor` 向共享 `llm_thread_pool` 提交 LLM 调用
- **输出**：每个任务实际并发数稳定收敛到 `pool_size / N`，无 starvation；窗口变化（任务进出）后能在一个 LLM 调用时长内重新平衡

## 详细规则

### 替换实现：手写 `AdjustableLimiter`

不再使用 `asyncio.Semaphore`，改用以下结构：

```python
class AdjustableLimiter:
    """可动态调整容量的并发限制器，不会因容量变化产生 permit 泄漏。"""

    def __init__(self):
        self._in_flight = 0
        self._capacity = 0
        self._waiters: list[asyncio.Future] = []
        self._loop: asyncio.AbstractEventLoop | None = None

    def set_capacity(self, capacity: int) -> None:
        """更新目标容量。capacity 增大时唤醒等待者；缩小不撤回已发出的 slot。"""

    async def acquire(self) -> None:
        """阻塞直到 in_flight < capacity，然后 in_flight += 1。"""

    def release(self) -> None:
        """线程安全减计数并唤醒一个等待者（通过 call_soon_threadsafe）。"""
```

`SlidingWindowExecutor` 内部持有一个 `AdjustableLimiter`，`acquire()` 时先 `set_capacity(_window_size())` 再 `await limiter.acquire()`。`track()` 注册的回调改为 `loop.call_soon_threadsafe(limiter.release)`。

### 关键不变量

1. **`in_flight` 永远等于该 executor 已 acquire 且未 release 的 slot 数**：不会因为对象重建而错乱
2. **`set_capacity()` 只更新目标值**，不撤回已经发出去的 slot：缩小时让自然完成消化，避免阻塞或撤销
3. **`acquire()` 检查的是 `in_flight >= capacity`**：每次进入都基于当前真实状态，不会因为缓存的 permit 数偏离
4. **`release()` 必须通过 `call_soon_threadsafe`**：done_callback 可能在 worker 线程触发，不能直接修改事件循环内的状态

### 公平性保证

- User A 单跑：`active=1`、capacity=40，`in_flight` 涨到 40 后 acquire 阻塞 ✓
- User B 进来：`active=2`，A 的 capacity 立刻变 20。A 当前 `in_flight=40 >= 20`，A 的下一次 acquire 阻塞，**不再有"突击 20 个"**
- A 完成一个：`in_flight=39`，但 `39 >= 20` 仍然 ≥ capacity，A 不能再提交。直到 `in_flight` 跌到 19 才放新的进来
- B 提交到池的任务排在队头（因为 A 不再往队尾塞新的），下一个空 worker 直接跑 B
- 稳态：A 占 20、B 占 20，分配真正公平

### 信号量重建的回归保护

`SlidingWindowExecutor` 接口不变（`acquire / track / iter_done / drain / pending_count` 全部保留），7 个 router 完全不需改动。

## UI / 接口变更

无。纯后端调度修复，前端与 API 完全无感。

## 测试方案

新建 `backend/tests/test_thread_pool_fairness.py`，至少覆盖：

1. **单任务独占**：active=1 时窗口=40，可同时提交 40 个并阻塞第 41 个
2. **窗口缩小**：active=1→2 后，旧任务从 40 in_flight 平滑降到 20，期间不会"突击提交"
3. **窗口扩大**：active=2→1 后，剩余任务的 capacity 升到 40，能继续提交至 40
4. **公平性**：模拟 A 已占满 40，B 加入后跑 30s，断言 A、B 完成数比例在 [0.8, 1.25] 之间（公平 ±25%）
5. **无 permit 泄漏**：窗口反复缩放（40→20→40→20）100 次后，单任务实际并发严格 ≤ capacity

**测试不依赖真实 LLM**：worker 函数用 `time.sleep(模拟耗时)` 模拟 LLM 调用，纯本地运行。本机环境暂时跑不通真实 LLM，测试只覆盖调度逻辑本身。

## 部署 / 重启策略

- 修复后需重启 `uvicorn`/`backend` 服务才能生效
- 当前正在运行的 3 个任务（XF / ZY / DYY）建议**先让用户在前端暂停**，重启后用各自的"恢复任务"按钮续跑；或等 DYY 跑完后再重启（取决于运维节奏，需用户决定）

## 不做的事

- 不改 router 层代码
- 不引入 per-user 优先级或 QoS（按当前 active_task_count 平均分配即可）
- 不动 `ThreadPoolExecutor` 本身（`max_workers=40` 不变）
- 不改前端
