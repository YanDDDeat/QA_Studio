# QA Studio 新手使用 SOP

## 1. 文档目的

本文面向第一次使用 QA Studio 的新用户，说明系统背景、核心概念、操作前准备、标准处理流程、各页面用途、常见问题和交付检查项。

读完本文后，用户应能独立完成一条从原始文本 JSON 到最终 QA 数据集的生产流程。

---

## 2. 系统背景

QA Studio 是一个面向高质量 QA 数据生产的工作台。它把原始文本资料转换为结构化问答数据，并支持知识体系生成、问题校验、答案生成、答案校验、数据评分、COT 过滤、数据集切分等后处理能力。

典型输入来源包括：

- 图书、文献、专利等资料经 PDF/Markdown 解析后切分出的文本块
- 已整理好的 JSON 文本片段
- 已经包含部分字段的半成品 QA JSON

典型输出是一个可交付的 JSON 数据集。每条记录通常包含问题、答案、推理链、来源、知识体系、难度、评分等字段。

---

## 3. 核心原则

### 3.1 文件是主线

系统以 JSON 文件为主线流转。用户上传或生成的文件都会进入系统文件管理区。后续阶段通常选择上一阶段产出的 JSON 文件继续处理。

### 3.2 每个阶段都是独立任务

问题生成、知识体系生成、问题校验、答案生成、答案校验、数据评分等页面都以“任务”的形式运行。

任务运行后可以查看：

- 任务进度
- 任务日志
- 输出文件
- 失败原因或异常信息

### 3.3 文本预处理已独立

原始文本里的目录、参考文献、图片链接、表格残骸、过短 chunk 等内容会影响问题质量。因此系统提供了独立的“文本预处理”工具。

推荐流程是：

原始 JSON → 文本预处理 → 预处理后 JSON → 问题生成

这样问题生成的输入文件是固定的，暂停、恢复、重试都不会受预处理参数变化影响。

---

## 4. 使用前准备

### 4.1 账号

系统无开放注册。账号由管理员创建。

首次使用前确认：

- 已获得用户名和密码
- 可以正常登录
- 登录后只会看到自己上传或生成的数据

### 4.2 LLM 配置

进入“配置中心”检查 LLM 配置。

需要确认：

- Base URL 正确
- API Key 可用
- 模型列表包含要使用的模型
- 默认模型可正常调用

可在 LLM 配置列表中点击“测试连接”验证。

### 4.3 Prompt 配置

进入“配置中心 → 提示词配置”，确认各阶段 Prompt 已配置。

重点检查：

- 问题生成 Prompt
- 知识体系生成 Prompt
- 问题校验 Prompt
- 答案生成 Prompt
- 答案校验 Prompt
- 数据评分 Prompt

如果需要修改 Prompt，应保存为新版本，避免覆盖历史版本。

### 4.4 输入文件格式

推荐上传 JSON 数组，每条记录是一段文本：

```json
[
  {
    "text": "RDX为白色粉末状结晶，具有较高爆速和爆压...",
    "source": "单质与混合火工药剂",
    "source_id": "book_001_chunk_028"
  }
]
```

常见字段说明：

| 字段 | 是否必需 | 说明 |
|---|---:|---|
| `text` | 是 | 原始文本内容，上传时可指定其他字段名 |
| `source` | 否 | 来源名称，如书名、文献名 |
| `source_id` | 否 | 来源定位 ID，如章节、chunk 编号 |

## 5. 标准生产流程

### 阶段 0：上传或选择原始文件

入口：

“文本预处理”或“文件管理”

操作：

1. 上传 JSON 文件。
2. 指定文本字段名，默认是 `text`。
3. 上传成功后，在文件列表中确认文件存在。

检查点：

- 文件记录数符合预期
- 文本字段选择正确
- 源文件预览能正常展示内容

---

### 阶段 1：文本预处理

入口：

“问题生成 → 文本预处理”

目的：

在正式调用 LLM 前，清理低质量文本块，减少 token 浪费和低质量问题。

系统会处理：

- 空内容
- 目录
- 参考文献
- 图片为主内容
- 表格残骸
- 低自然语言文本
- 过短文本合并或过滤
- 高频页眉页脚

操作：

1. 选择源 JSON 文件。
2. 选择文本字段。
3. 设置“最小 token”，默认 `200`。
4. 点击“开始预处理”。
5. 查看统计结果。
6. 加载“预处理结果预览”，确认清洗后的文本可用。

产出：

- `*_preprocessed.json`：清洗和合并后的文件，用于问题生成。
- `*_preprocess_filtered.json`：过滤明细文件，用于人工审查。

注意：

- 预处理结果已经保存到系统中，不需要在本页面下载。
- 后续在“问题生成”页面直接选择预处理后的 JSON 文件。
- 如需下载，去“数据中心”或“文件管理”下载。

检查点：

- 保留数量符合预期
- 跳过原因分布合理
- 预处理结果预览中没有明显目录页、参考文献堆、图片链接堆
- 过滤明细中没有大量误删的有效正文

---

### 阶段 2：问题生成

入口：

“问题生成 → 问题生成”

目的：

根据预处理后的文本生成 QA 问题记录。

操作：

1. 选择 `*_preprocessed.json` 文件。
2. 选择问题类别：
   - 知识问答
   - 逻辑生成
3. 选择来源类型：
   - 图书
   - 专利
   - 文献
   - 其他
4. 可选填写来源名称和来源 ID。
5. 选择问题生成 Prompt。
6. 选择 LLM 配置和模型。
7. 填写输出文件名。
8. 点击“开始生成”。

产出字段通常包括：

| 字段 | 说明 |
|---|---|
| `input` | LLM 生成的问题 |
| `category` | 用户选择的问题类别 |
| `task_type` | LLM 返回的任务类型 |
| `domain` | LLM 返回的领域 |
| `difficulty` | LLM 返回的难度 |
| `originContent` | 来源文本 |
| `source` | 来源名称 |
| `source_id` | 来源 ID |
| `source_type` | 来源类型 |

检查点：

- 任务日志无连续调用失败
- 输出文件已生成
- 问题内容和来源文本匹配
- 没有明显由目录、参考文献生成的问题

---

### 阶段 3：知识体系生成

入口：

“问题生成 → 知识体系生成”

目的：

为每条问题记录补充知识体系字段。

操作：

1. 选择问题生成阶段输出文件。
2. 选择知识体系生成 Prompt。
3. 选择 LLM 配置和模型。
4. 点击“开始生成”。

产出字段：

| 字段 | 说明 |
|---|---|
| `knowledge` | 知识体系路径或标签 |
| `scene` | 场景字段，通常与知识体系相关 |

检查点：

- `knowledge` 字段不为空
- 分类粒度符合业务要求
- 没有大面积跑偏或格式异常

---

### 阶段 4：问题校验

入口：

“问题生成 → 问题校验”

目的：

筛除低质量问题。

操作：

1. 选择知识体系生成后的文件。
2. 选择问题校验 Prompt。
3. 选择 LLM 配置和模型。
4. 点击“开始生成”。

系统逻辑：

- PASS 的记录进入下一阶段。
- FAIL 的记录会写入单独失败文件，供人工查看。

检查点：

- 通过率是否合理
- 失败文件是否存在
- 抽查 FAIL 原因是否可信
- 不应出现大量好问题被误杀

---

### 阶段 5：答案生成

入口：

“答案生成 → 答案生成”

目的：

为通过校验的问题生成答案和推理链。

操作：

1. 选择问题校验后的文件。
2. 选择答案生成 Prompt。
3. 选择 LLM 配置和模型。
4. 点击“开始生成”。

产出字段：

| 字段 | 说明 |
|---|---|
| `output` | 答案 |
| `cot` | 推理过程或解题过程 |

检查点：

- 答案能直接回应问题
- 推理链与答案一致
- 没有明显编造来源文本之外的信息

---

### 阶段 6：答案校验

入口：

“答案生成 → 答案校验”

目的：

筛除错误、空泛、幻觉或格式不合格的答案。

操作：

1. 选择答案生成后的文件。
2. 选择答案校验 Prompt。
3. 选择 LLM 配置和模型。
4. 点击“开始生成”。

系统逻辑：

- PASS 的记录保留。
- FAIL 的记录写入答案校验失败文件。

检查点：

- 失败原因是否明确
- 高风险答案是否被筛掉
- 通过记录的答案质量是否稳定

---

### 阶段 7：数据评分

入口：

“数据评分 → 数据评分”

目的：

对最终 QA 数据进行维度评分。

操作：

1. 选择答案校验后的文件。
2. 选择数据评分 Prompt。
3. 选择 LLM 配置和模型。
4. 点击“开始生成”。

产出字段：

| 字段 | 说明 |
|---|---|
| `relevance` | 相关性 |
| `clarity` | 清晰度 |
| `reasoning` | 推理质量 |
| `terminology` | 术语准确性 |
| `score` | 综合评分 |

检查点：

- 评分字段完整
- 低质量数据分数明显偏低
- 评分分布符合预期

---

### 阶段 8：质检、COT 过滤与数据集切分

这些步骤按项目交付要求选择使用。

#### 数据质检

入口：

“数据质检 → 质检”

用途：

对已生成数据做额外质量检查，适合交付前抽检或规则补充检查。

#### COT 过滤

入口：

“数据后处理 → COT过滤”

用途：

将带 COT 和不带 COT 的数据拆分或筛选，满足不同训练格式要求。

#### 测试集切分

入口：

“数据后处理 → 测试集切分”

用途：

按比例或规则拆分训练集、测试集等。

---

## 6. 数据查看、下载与交付

### 6.1 数据中心

入口：

“管理中心 → 数据中心”

用途：

- 查看文件内记录
- 查看字段详情
- 按任务类型、领域、难度等过滤
- 检查最终数据质量

### 6.2 文件管理

入口：

“管理中心 → 文件管理”

用途：

- 查看系统内全部文件
- 下载 JSON 文件
- 删除不需要的文件
- 同步或修复文件内容

### 6.3 交付前检查

交付前建议确认：

- 文件记录数符合预期
- 关键字段完整：`input`、`output`、`cot`、`knowledge`、`source`、`source_id`
- 校验失败文件已单独保存
- 评分字段完整
- 抽样记录不存在明显幻觉、错答、格式错乱
- 下载的是最终阶段文件，而不是中间文件

---

## 7. 任务暂停、恢复与重试

### 7.1 暂停

任务运行中可点击“停止”。系统会尽量在当前记录处理完成后暂停，并把已完成结果写入文件。

### 7.2 恢复

暂停任务可恢复。恢复时系统从已完成进度继续。

因为文本预处理已经独立为固定文件，问题生成恢复时不会重新计算过滤结果，进度更稳定。

### 7.3 重试

失败任务可重试。

重试前建议先查看任务日志，判断失败原因：

- LLM 接口异常
- API Key 不可用
- 模型不可用
- Prompt 返回格式不符合 JSON
- 输入文件字段缺失

## 8. 常见问题

### 8.1 预处理后有效文本变少怎么办？

降低“最小 token”阈值，或查看过滤明细，判断是否误删。

如果大量有效短文本被过滤，可以：

- 降低阈值
- 调整原始 chunk 切分方式
- 合并原始文本后重新上传

### 8.2 问题生成没有结果怎么办？

检查：

- 输入文件是否选成了过滤明细文件
- 文本字段是否正确
- Prompt 是否要求返回 JSON
- LLM 配置是否可用
- 任务日志是否有连续调用失败

### 8.3 后续阶段找不到合适文件怎么办？

进入文件管理或数据中心确认上一阶段是否成功生成输出文件。

也可以打开“显示全部文件”或直接在文件管理中查看文件来源。

### 8.4 生成质量不稳定怎么办？

优先检查：

- 原始文本质量
- 是否经过文本预处理
- Prompt 是否过宽泛
- 模型是否适合当前任务
- 校验 Prompt 是否足够严格

### 8.5 如何避免重复处理？

建议按文件名保留阶段信息，并在每个阶段完成后使用该阶段输出文件继续。

不要在多个相似文件之间来回切换，除非明确知道每个文件的来源。

---

## 9. 推荐命名规范

建议输出文件名包含资料名、阶段和批次信息。

示例：

```text
RDX资料_预处理_batch01
RDX资料_问题生成_batch01
RDX资料_知识体系_batch01
RDX资料_答案生成_batch01
RDX资料_最终评分_batch01
```

系统会自动追加用户名、时间戳和必要后缀。

---

## 10. 新人最短路径

如果只想快速跑通一条完整链路，按下面顺序执行：

1. 登录系统。
2. 在“配置中心”确认 LLM 配置和 Prompt。
3. 进入“文本预处理”，上传原始 JSON。
4. 设置文本字段和最小 token，执行预处理。
5. 查看预处理结果预览。
6. 进入“问题生成”，选择预处理后的文件，生成问题。
7. 进入“知识体系生成”，选择问题生成文件。
8. 进入“问题校验”，筛除低质量问题。
9. 进入“答案生成”，生成答案和 COT。
10. 进入“答案校验”，筛除低质量答案。
11. 进入“数据评分”，生成评分字段。
12. 在“数据中心”抽查数据。
13. 在“文件管理”下载最终 JSON。

---

## 11. 关键注意事项

- 文本预处理不是最终交付阶段，但强烈建议先做。
- 问题生成应优先选择 `*_preprocessed.json` 文件。
- 过滤明细文件只用于审查，不应作为问题生成输入。
- 每次任务前确认 Prompt、模型、输入文件都正确。
- 任务失败先看日志，不要盲目重复点击。
- 最终交付前必须抽样检查，不要只看任务成功状态。
